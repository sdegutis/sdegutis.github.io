First thoughts on Haskell I spent the past few weeks using the evening and weekend to learn Haskell and research how I could use it to build a web app. Here's the most interesting points I found: ### Compared to Clojure 1. Rich borrowed almost all of Clojure's standard library from Haskell. 2. Destructuring in Clojure also looks like it came directly from Haskell. 3. Haskell's pattern matching is so much more useful than Clojure's core.match library. 4. Haskell's sum types are really indispensable, I really wish Clojure had them. 5. Static typing acts like free unit tests! 6. Type inference means you almost never have to specify a type, so it feels as limber as dynamic typing. ### Neat features 1. You can use the `$` operator to avoid parentheses. So `foo quux (bar baz)` can become `foo quux $ bar baz`. 2. All functions are curried, so there's no need for Clojure's `partial` function. 3. You can make your own operators, with their own precedence and associativity. 4. Type classes are great for polymorphism, and can also define operators (like Eq and Ord do). ### Uses for monads 1. They make sequential operations much less ugly while working in a purely-functional language. 2. They can be used to separate interface from implementation. 3. They make error propagation much simpler. In almost all cases, you can use the Either or Maybe monads to replace exceptions. 4. A lot of what they call EDSLs (like Clay, Lucid, Scotty, etc.) are made using monads. ### The bad 1. I love Haskell's syntax in theory, but real-world code is kind of hard for me to read. Maybe it just takes time like it did with Clojure. 2. We have to pay attention to operator precedence once again. 3. The tooling around Haskell isn't as good as in the Clojure ecosystem, but it's not that far off. 4. The "do" syntactic sugar for chaining monads can trick you into thinking you're writing imperative code. This confused me at first. ### The ugly 1. Records (like `defrecord`) can't use a field name that's already been used within another record in the same namespace. 2. Monads still confuse me a lot, even though I think I understand them. 3. There's such a thing as "Monad Transformers" which sound even scarier than Monads to me. I've yet to even look into them. 4. Compiling a Haskell project is about as slow as compiling a Clojure project. 5. I really don't like lazy IO. But it's not related to non-strict evaluation, so it's very much an opt-in feature that I just wouldn't use. ### Some interesting libraries 1. [Clay](http://fvisser.nl/clay/): library to generate CSS, like Garden. 2. [Lucid](https://github.com/chrisdone/lucid): library to generate HTML, like Hiccup. 3. [Hspec](http://hspec.github.io/): RSpec clone. 4. [Scotty](https://github.com/scotty-web/scotty): just like Compojure, built on top of [WAI](http://www.yesodweb.com/book/web-application-interface), the Haskell equivalent to Ring. 5. [Aeson](https://hackage.haskell.org/package/aeson-0.6.1.0/docs/Data-Aeson.html): JSON library, but serves a similar purpose as [Prismatic's Schema](https://github.com/Prismatic/schema). There are also a few Haskell-to-JavaScript compilers, but the situation's still in flux. Some people want a full runtime, others want none. Some want lazy ("non-strict") evaluation, some want normal evaluation so that it emits more readable JavaScript. Anyway, just some thoughts on Haskell. It looks really cool, but I'm still undecided on whether I want to actually write a real-world app in Haskell.
