First thoughts on OCaml I forgot a lot of the reasons I originally didn't like about OCaml and started learning Haskell. I should have started taking notes sooner. Anyway, here are my new notes as I re-learn OCaml. ### Great 1. Great support for FP functionality just like Clojure. 2. Destructuring everywhere, just like in Clojure! (They call it pattern matching.) 3. Not whitespace-sensitive! 4. Variant (sum) types! 5. No need for the IO monad! Pragmatic about side-effects, like Clojure. 6. Super fast! Even when compiled to JS! ### Undecided 1. No type classes. 2. Syntax isn't as pure/terse as Haskell's. 3. I've heard good things about OPAM. ### Unfair but true 1. [Real World OCaml](https://realworldocaml.org/) is really really slow; can't find faster-paced book. 2. [js\_of\_ocaml's website](http://ocsigen.org/js%5Fof%5Focaml/) is both very ugly and hard to navigate. 3. [Official Manual](http://caml.inria.fr/pub/docs/manual-ocaml/) is both ugly and seems to skip valuable information. ### Initially didn't like, but don't mind anymore 1. Need to put abstract interface into its own `.mli` file. 2. Existence of keyword-based arguments. ### Open questions 1. Is OPAM an equivalent to Bundler and/or Leiningen? 2. Does it share Haskell's "section" feature? ### Answered questions 1. Does it share Haskell's record field name collision problem? **nope!** 2. Where do you search for libraries? **opam** 3. How do you install libraries? **opam** ### Gross 1. Difficult FFI with C, even when using a third party library. 2. Heard bad things about Jane Street's `Core` library. 3. Ugly syntax for keyword-based arguments. ### Resources 1. [OCaml for Haskellers](http://blog.ezyang.com/2010/10/ocaml-for-haskellers/) 2. [using ObjC from OCaml](http://psellos.com/2014/12/2014.12.objc-rule-four.html) 3. [one alternative to Core](http://seb.mondet.org/software/nonstd/index.html)
