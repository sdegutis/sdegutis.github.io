Making your own tools I[recently wrote](/blog/2013-11-21-starting-from-scratch)about my discovery that reducing a problem's incidental complexity reveals how simple it really is and makes it much more approachable. A really fun application of this principle is making your own tools. In that blog post, I listed the ones I made in 2013, most of which I use on a daily basis. This works best when you're making tools that don't depend on outside adoption. Tools that you directly benefit from, even if you're the only user. Some tools you could replace: * Text editor * Email client * Web browser * Programming language * Window manager * Music player * Terminal + shell + command line utilities[\[1\]](#fn1) Some of these may seem scary, or like they're too big to replace. But don't panic! Just remember these simple laws of physics: You probably don't use the majority of the features it provides anyway, so you can just omit those features. And the features you do use, you probably already know how to write. When you finally hit a feature that actually is beyond you, there's almost guaranteed to be a free[\[2\]](#fn2) library[\[3\]](#fn3) for[\[4\]](#fn4) it[\[5\]](#fn5) on[\[6\]](#fn6) the[\[7\]](#fn7) internet[\[8\]](#fn8). When that turns out empty results and you're genuinely stuck on a hard feature, just switch over to working on the easier ones. When all that's left is the hard one, it'll suddenly look a whole lot easier. Plus, that gives you more time to mull it over; the answer often comes from nowhere when you're waking up or cooking dinner. Whatever the tool is that you're replacing, even if you're the only one using it, if it makes you at least a little more productive than the alternatives, you'll be reaping benefits for years to come. --- 1.
