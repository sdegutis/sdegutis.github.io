First thoughts on front-end tools Front-end development isn't much more complex than when I first started learning it in the late 90s. But people have made some really good tools that make it a lot easier to get started and to build larger, more modular front-end apps. So I've been researching them and here's what I've got so far: ### Build tools [Gulp](http://gulpjs.com) | a simple streaming build tool I've used and liked in the past.[Browserify](http://browserify.org) | for writing modular code.[Bower](http://bower.io) | not yet sure why I'd use this over NPM...[Babel](https://babeljs.io) | Let's me use ES6 features today! (Destructuring, arrow functions, etc.) linters | [eslint](http://eslint.org) \+ [jscs](https://github.com/jscs-dev/node-jscs) \+ [jshint](http://jshint.com) \- "i use all 3\. gives me amazing coverage :-)" (says ljharb)[UglifyJS2](https://github.com/mishoo/UglifyJS2) | to compress compiled JS source file.[es6-tools](https://github.com/addyosmani/es6-tools) | really just a reference for hooking up the other things here. ### JS libs [Mithril](http://lhorie.github.io/mithril/) | For dynamic generation of the DOM (i.e. the stuff that makes it interactive).[lodash](https://lodash.com) | like Underscore.js but better and more cross-platform-ish; lets me pretend I'm using ClojureScript. Shims | [es5-shim](https://github.com/es-shims/es5-shim), [es6-shim](https://github.com/paulmillr/es6-shim/), [es7-shim](https://github.com/es-shims/es7-shim) \- someone told me this effectively obsoletes lodash, and to "configure babel to not transpile away the things that es6-shim/es7-shim provides" because "core-js doesn't fix browser incompatibilities" ### CSS stuff [Grid](http://adamkaplan.me/grid/) | absolutely indispensable information, including a grid concept for "responsible" design.[Stylus](http://learnboost.github.io/stylus/) | Because CSS by itself is awful.[Normalize.css](http://necolas.github.io/normalize.css/) | To significantly reduce per-browser CSS quirks.[kouto-swiss](http://kouto-swiss.io/docs.html) | reduces CSS boilerplate for common and useful things. Best practices | OOCSS, ACSS, BEM, and SMACSS -- still don't know much about these yet. ### Might need one day [Jade](http://jade-lang.com) | Nicer HTML templating. Mithril already solves this for me for the most part.[Handlebars](http://handlebarsjs.com) | Non-HTMl templating. Don't need yet but might? Superior to Mustache. ### Why Mithril instead of React? 1. It's smaller (literally under 1k heavily commented lines of code). 2. It doesn't require JSX, just do `m("div", {onclick: foo}, "bla")`. 3. It lets you re-use simple JavaScript idioms instead of inventing its own. 4. It's _sooo_ much simpler than React, while doing the same thing. 5. Actually no, it does _even more_ than React, since React is only V but Mithril is MVC. 6. All while still being smaller and simpler. Also, I just learned yesterday that [MELPA now uses Mithril](https://github.com/milkypostman/melpa/blob/master/html/js/melpa.js). ### Why Browserify instead of Webpack?
